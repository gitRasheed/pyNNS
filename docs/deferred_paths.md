# Deferred Paths

Deferred paths include explicit `NotImplementedError` guards plus public API
surfaces that remain blocked by those guarded paths. Keep the dependency notes
current so temporary scope limits stay visible.

| Area | Deferred path | Reason | Dependency / next action |
|---|---|---|---|
| ARMA.optim | default optimizer path | Default optimizer evaluates `nns_reg(..., smooth=True)`, so it cannot be faithfully ported until smooth regression is real | Port `smooth=True` first, then `NNS.ARMA.optim` |
| Boost | `threshold` on the `n_features > 10` stochastic path | The ordinary stochastic epoch keeper path, including `ts_test`, is supported structurally with PyNNS RNG. Installed R errors for `threshold=` because `test.features` is never built. | Keep threshold guarded to match installed-R error behavior |
| Boost | broad factor predictor edge cases | Simple single-column explicit-level factor predictors are supported by R-style integer coding before deterministic feature selection. Multiple explicit-level factor predictors remain deferred: installed-R probes can produce matching feature diagnostics in one case but diverge in another, final predictions also diverge, and another probe errored inside R `NNS.reg`. | Map R final-estimate behavior, feature-set scoring diagnostics, and error boundaries before claiming broad factor-predictor parity |
| Differentiation | point-evaluated derivative wrappers | `dy_dx(..., eval_point="overall")` is supported because installed R computes it from existing unsmoothed `NNS.reg` fitted gradients. Numeric `dy.dx` evaluation points and all `dy.d_` paths call `NNS.reg(..., smooth=True)` for finite-difference estimates, so spline-eligible inputs require the same smooth-spline backend as smooth regression. | Port `smooth=True`, then enable point-evaluated `dy_dx` and `dy_d` finite-difference wrappers |
| Regression | `confidence_interval` with `smooth=True` for spline-eligible fits | R only calls `stats::smooth.spline(..., spar=(dependence + 0.5) / 2)` when the regression-point count is at least 4 and `order` is not character; smaller univariate inputs fall back to the ordinary piecewise path and are supported. Spline-eligible interval tables require an R-compatible fixed-spar smoothing spline backend. | Port a minimal R-compatible fixed-spar smoothing spline backend |
| Regression | spline-eligible `smooth=True` | R only calls `stats::smooth.spline(..., spar=(dependence + 0.5) / 2)` when the regression-point count is at least 4 and `order` is not character; smaller univariate inputs fall back to the ordinary piecewise path and are supported. SciPy smoothers are not parity-compatible. | Port a minimal R-compatible fixed-spar smoothing spline backend, then enable spline-eligible `nns_reg(smooth=True)` |
| Stack | factor predictor method-2 diagnostics | `nns_stack` method 1 supports explicit `factor_levels`, but factor predictors with method 2 are rejected. Installed-R probes show unstable method-2 behavior: one factor-only method-2 case errors on row counts, while a mixed factor/numeric case returns matching dim-red predictions but different method-2 diagnostics and stacked method-1 internals from the current PyNNS expansion path. | Map R's factor expansion, method eligibility, diagnostic field assignment, and stacked method-1 interaction before broadening factor method-2 parity claims |
| Multivariate regression | direct `factor_2_dummy=True` raw predictor path | Public `nns_reg` factor predictors are supported, but installed R direct `NNS.M.reg` raw factor input errors; PyNNS keeps direct `nns_m_reg` raw factor input rejected unless another public R path requires it | Keep rejected; revisit only if a public installed-R path depends on direct `NNS.M.reg` factor expansion |
| VAR | default VAR path | Depends on `NNS.ARMA.optim` and `nns_stack(ts_test)`; `ts_test` is done, but `NNS.ARMA.optim` remains blocked by smooth regression | Port `smooth=True`, then `NNS.ARMA.optim`, then `NNS.VAR` |
| Nowcast | nowcast wrapper | Installed R fetches external macroeconomic series and delegates to `NNS.VAR`; PyNNS cannot faithfully expose the wrapper until `nns_var` is real, and it should avoid implicit runtime data-provider side effects until that API is designed | Port `NNS.VAR`, then design an explicit data-ingestion boundary for `nns_nowcast` |
